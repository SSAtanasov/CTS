## Тема 6: Списъци за контрол на достъпа (ACL)

**Цел:** Внедряване на ACL за филтриране на трафика и намаляване на мрежовите атаки

---

### Насоки за конфигуриране на ACL

ACL се състои от един или повече записа за контрол на достъпа (ACE). При конфигуриране и прилагане се спазват следните насоки:

- ACL се създава глобално и след това се прилага към интерфейс или линия.
- Последното правило е неявно **„забрани всичко"** (`deny any` / `deny ip any any`).
- Редът на правилата е от значение — ACL се обработва **отгоре надолу**; при съвпадение обработката приключва незабавно.
- Филтрирайте от **най-конкретното към най-общото** — например: забранете конкретен хост, след което разрешете всички останали.
- На интерфейс се допуска само **един ACL** за всеки протокол и посока.
- Новите правила към съществуващ ACL се добавят в **края** му по подразбиране.
- Пакетите, генерирани от самия маршрутизатор, **не се филтрират** от изходящи ACL.
- **Стандартните ACL** се поставят възможно най-близо до **получателя**.
- **Разширените ACL** се поставят възможно най-близо до **източника**.

---

### Прилагане на ACL

ACL се прилага към интерфейс или VTY линии чрез:

```
Router(config-if)# ip access-group {acl-# | name} {in | out}
Router(config-line)# ip access-class {acl-# | name} {in | out}
```

**Именуван стандартен ACL — изходящ трафик:**
```
R1(config)# ip access-list standard NO_ACCESS
R1(config-std-nacl)# deny host 192.168.11.10
R1(config-std-nacl)# permit any
R1(config)# interface g0/0
R1(config-if)# ip access-group NO_ACCESS out
```

**Именуван разширен ACL — входящ и изходящ трафик:**

ACL `SURFING` се прилага входящо, `BROWSING` — изходящо на интерфейс G0/0:

```
R1(config)# ip access-list extended SURFING
R1(config-ext-nacl)# permit tcp 192.168.10.0 0.0.0.255 any eq 80
R1(config-ext-nacl)# permit tcp 192.168.10.0 0.0.0.255 any eq 443
R1(config)# ip access-list extended BROWSING
R1(config-ext-nacl)# permit tcp any 192.168.10.0 0.0.0.255 established
R1(config-if)# ip access-group SURFING in
R1(config-if)# ip access-group BROWSING out
```

**Именуван ACL на VTY линии с регистрация:**

Параметърът `log` регистрира всяко съвпадение, но натоварва устройството — да се използва само при активна атака за идентифициране на нападателя.

```
R1(config)# ip access-list standard VTY_ACCESS
R1(config-std-nacl)# permit 192.168.10.10 log
R1(config-std-nacl)# deny any
R1(config)# line vty 0 4
R1(config-line)# access-class VTY_ACCESS in
```

ACL са неразделна част и от NAT, ZPF и VPN конфигурациите.

Примерен изход след прилагане на ACL с log:
```
*Feb 26 18:58:30.579: %SEC-6-IPACCESSLOGNP: list VTY_ACCESS permitted 0
192.168.10.10 -> 0.0.0.0, 5 packets

R1# show access-lists
Standard IP access list VTY_ACCESS
   10 permit 192.168.10.10 log (6 matches)
   20 deny any
```

За премахване на ACL от интерфейс: `no ip access-group`. За пълно изтриване от маршрутизатора: `no access-list`.

---

### Къде да се поставят ACL

Всеки ACL трябва да е поставен там, където е най-ефективен. Стандартните и разширените ACL имат различни правила за поставяне.

Поставянето и типът на ACL зависи и от следните фактори:
- Обхватът на административния контрол на организацията върху устройствата
- Пропускателната способност на мрежите между устройствата
- Лекотата на конфигуриране

Ако стандартен ACL се постави при **източника**, правилото ще се приложи само въз основа на изходящия адрес — независимо от крайната дестинация. Затова стандартните ACL се поставят **близо до получателя**.

---

### Пример за поставяне на стандартен ACL

Цел: забрани трафика от `192.168.10.0/24` към `192.168.30.0/24`. Стандартните ACL се поставят **близо до получателя** — на R3.

- **R3 S0/1/1 (входящ)** — не е подходящ: би блокирал и трафика към `192.168.31.0/24`.
- **R3 G0/0/0 (изходящ)** — **правилното място**: блокира само трафика към `.30`, без да засяга `.31`.

### Пример за поставяне на разширен ACL

Разширените ACL се поставят **близо до източника**, за да се предотврати пренасянето на нежелан трафик из цялата мрежа. ACL може да се постави само на устройства под контрол на организацията.

**Пример:** Компания А иска да забрани Telnet и FTP от `192.168.11.0/24` към `192.168.30.0/24`.

- **R1 S0/1/0 (изходящ)** — не е оптимален: обработва всички пакети, напускащи R1.
- **R1 G0/0/1 (входящ)** — **правилното място**: само пакетите от `192.168.11.0/24` се обработват от ACL — минимално натоварване, максимална точност.

## Намаляване на атаките чрез ACL

### Намаляване на атаките чрез подправяне

Повечето DoS атаки използват подправяне на IP адреса. Нападателят вмъква произволен изходящ IP адрес, за да скрие самоличността си.

На интерфейса към интернет (S0/0/0) никога не трябва да влизат пакети от:

```
R1(config)# access-list 150 deny ip host 0.0.0.0 any
R1(config)# access-list 150 deny ip 10.0.0.0 0.255.255.255 any
R1(config)# access-list 150 deny ip 127.0.0.0 0.255.255.255 any
R1(config)# access-list 150 deny ip 172.16.0.0 0.15.255.255 any
R1(config)# access-list 150 deny ip 192.168.0.0 0.0.255.255 any
R1(config)# access-list 150 deny ip 169.254.0.0 0.0.255.255 any
R1(config)# access-list 150 deny ip 224.0.0.0 15.255.255.255 any
```

На вътрешния интерфейс (G0/0) се разрешават само пакети с изходящ адрес от локалната мрежа — всички останали се отхвърлят.

---

### Разрешаване на необходимия трафик през защитна стена

Изрично се разрешават само необходими услуги. Чести примери: DNS, SMTP, FTP за достъп отвън; SSH, syslog, SNMP за административен отдалечен достъп. Всички тези услуги трябва да се контролират — злоупотребата с тях води до уязвимости.

Пример за ACL входящо на S0/0/0:

```
R1(config)# access-list 180 permit udp any host 192.168.20.2 eq domain
R1(config)# access-list 180 permit tcp any host 192.168.20.2 eq smtp
R1(config)# access-list 180 permit tcp any host 192.168.20.2 eq ftp
R1(config)# access-list 180 permit tcp host 200.5.5.5 host 192.168.20.2 eq ssh
```

---

### Намаляване на ICMP атаките

Нападателите използват ICMP echo пакети за разкриване на хостове и DoS атаки, а ICMP redirect — за промяна на маршрутни таблици. Входящите echo и redirect трябва да се блокират.

**Разрешени входящо:**
- `echo-reply` — позволява ping до външни хостове
- `source-quench` — иска намаляване на трафика
- `unreachable` — генерира се при административно забранени пакети

**Разрешени изходящо:**
- `echo` — позволява ping навън
- `parameter-problem` — информира за проблеми в заглавката
- `packet-too-big` — MTU discovery
- `source-quench` — ограничава трафика

```
R1(config)# access-list 114 permit icmp 192.168.1.0 0.0.0.255 any echo
R1(config)# access-list 114 permit icmp 192.168.1.0 0.0.0.255 any parameter-problem
R1(config)# access-list 114 permit icmp 192.168.1.0 0.0.0.255 any packet-too-big
R1(config)# access-list 114 permit icmp 192.168.1.0 0.0.0.255 any source-quench
```

> Всички останали типове ICMP изходящо се блокират.

---

### Намаляване на SNMP атаките

SNMP е полезен за мониторинг, но може да се злоупотреби. Мерки:
- Прилагане на интерфейсни ACL за филтриране на SNMP от неоторизирани системи. Внимание: подправен адрес може да заобиколи ACL.
- Ако SNMP не е необходим — деактивирайте го: `no snmp-server`.

## ACL за IPv6

### Общ преглед и примерна злоупотреба с уязвимост

IPv4 и IPv6 ще съществуват едновременно, което създава потенциални дупки в сигурността. В среди с **двоен стек (dual stack)** нападателите могат да използват IPv4 за атака на IPv6.

**Механизъм на атаката:**
1. Нападателят получава опора в IPv4 мрежата.
2. Компрометираният хост изпраща фалшиви Router Advertisements (RA), карайки двустекови хостове да получат IPv6 адрес.
3. Нападателят се движи странично из мрежата, компрометира допълнителни хостове и изнася трафик навън.

**Teredo tunneling** — IPv6 преходна технология, вграждаща IPv6 пакети в IPv4 UDP пакети за преминаване през NAT устройства. Нападателите я използват за скрити атаки.

---

### Конфигуриране на ACL за IPv6

IPv6 ACL съдържа неявно `deny ipv6 any any` и неявни правила за NDP (Neighbor Discovery Protocol) — NA и NS пакети. Ако `deny ipv6 any any` се конфигурира без изрично разрешаване на NDP, протоколът за съседско откриване се деактивира.

**Добра практика:** Документирайте неявните правила изрично в конфигурацията.

```
R1(config)# ipv6 access-list LAN_ONLY
R1(config-ipv6-acl)# permit 2001:db8:1:1::/64 any
R1(config-ipv6-acl)# permit icmp any any nd-na
R1(config-ipv6-acl)# permit icmp any any nd-ns
R1(config-ipv6-acl)# deny ipv6 any any
```

## Тема 7: Защитни стени с базирана на зони политика (ZPF)

### Предимства на ZPF

Cisco IOS Firewall поддържа два модела:
- **Класическа защитна стена** — политиката се прилага директно на интерфейсите.
- **ZPF** — интерфейсите се присвояват към зони; политиката се прилага към трафика между зоните.

**Предимства на ZPF:**
- Независим от ACL.
- Стойността по подразбиране е **„блокирай, освен ако изрично е разрешено"**.
- Лесни за четене политики чрез **C3PL** (Cisco Common Classification Policy Language) — структурен метод, базиран на събития, условия и действия. Една политика обслужва даден трафик, вместо множество ACL.
- Виртуални и физически интерфейси могат да се групират в зони.
- Политиките са **еднопосочни** — между двойки зони.

> Ако допълнителен интерфейс се добави към вече съществуваща зона, хостовете на новия интерфейс автоматично могат да комуникират с хостовете на другите интерфейси в същата зона. Двата модела могат да работят едновременно, но **не** на един и същ интерфейс.

---

### Проектиране на ZPF

**Стъпка 1 — Определяне на зоните**
Зоните установяват границите на сигурност. Трафикът е подложен на ограничения при преминаване между зони. Пример: интернет = една зона, вътрешна мрежа = друга зона.

**Стъпка 2 — Установяване на политики между зоните**
За всяка двойка изходна/целева зона се дефинират допустимите сесии (TCP, UDP, ICMP). Политиките са **еднопосочни** и се дефинират въз основа на двойки зони (zone pairs).

**Стъпка 3 — Проектиране на физическата инфраструктура**
Определят се броят на устройствата между зоните и резервираността им.

**Стъпка 4 — Идентифициране на подмножества в зоните**
При сложни топологии множество зони могат да са свързани към един интерфейс — изискванията за трафик се обединяват в политика, специфична за устройството.

---

### Примери за ZPF топологии

- **LAN-to-Internet** — базова топология: вътрешна мрежа и интернет зона.
- **Защитна стена с публични сървъри (вариант 1 и 2)** — добавя DMZ зона за публично достъпни сървъри.
- **Резервирани защитни стени** — топология с две защитни стени за висока наличност.
- **Сложна защитна стена** — топология с множество зони и комплексни политики.

---

### Packet Tracer: Конфигуриране на ZPF

Практическо конфигуриране на ZPF на Cisco 1941 маршрутизатор с три зони: INTERNET (Serial 0/0/0), PRIVATE (GigabitEthernet 0/1) и DMZ (GigabitEthernet 0/0).

Работният процес включва:
1. Дефиниране на ACL за разрешен трафик между зоните
2. Създаване на зони: `zone security INTERNET`, `zone security PRIVATE`, `zone security DMZ`
3. Присвояване на интерфейси към зони: `zone-member security <ИМЕ>`
4. Конфигуриране на **class-map** — определя типа трафик за инспекция (по протокол или по ACL)
5. Конфигуриране на **policy-map** — присвоява class-map с действие (`inspect` / `pass` / `drop`)
6. Конфигуриране на **zone-pair** — свързва изходна и целева зона с policy-map чрез `service-policy type inspect`

> Веднага след присвояване на интерфейси към зони трафикът между тях се блокира по подразбиране — дотогава не са дефинирани политики.

**Class-map може да съвпада:**
- с трафик, дефиниран от ACL: `match access-group`
- с индивидуални протоколи: `match protocol http`, `match protocol ssh` и т.н.
- с друг class-map (вложени class-maps)

**Пример за три политики:**
- **Private → Internet**: инспектира HTTP, HTTPS, DNS, FTP, ICMP, SSH
- **Private → DMZ**: инспектира HTTP достъп до mail сървъра
- **Internet → DMZ**: разрешава достъп до уеб сървър (ACL 101, 192.168.2.252) и mail сървър (ACL 102, 192.168.2.253)

Class-maps се присвояват към policy-maps; policy-maps се присвояват към zone-pairs. Неприпокритият трафик (`class-default`) по подразбиране се отхвърля.
